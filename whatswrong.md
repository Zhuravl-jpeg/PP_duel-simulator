## ⚠️ Что НЕ готово для продакшена

### 1. **PostgreSQL пул не оптимизирован для serverless** 🔴 КРИТИЧНО

Файл `src/server/db/index.ts`:
```typescript
const pool = new pg.Pool({
  connectionString: process.env.DATABASE_URL,
});
```

**Проблема:** `pg.Pool` создаёт постоянные подключения. На Vercel/Railway (serverless) каждый cold start создаёт новый процесс, и пул не будет работать корректно — может превысить лимит соединений к БД.

**Решение:** Использовать `@neondatabase/serverless` или `pg` с `max` pooling + PgBouncer (Neon предоставляет встроенный).

---

### 2. **Нет CI/CD** 🔴 КРИТИЧНО

Папка `.github/` пуста — нет GitHub Actions workflow.

**Нужно создать:**
- `.github/workflows/ci.yml` — линтинг + тесты + сборка
- `.github/workflows/deploy.yml` — автодеплой на Vercel/Railway при мерже в main

---

### 3. **Better-auth не настроен** 🟡 ТРЕБУЕТ НАСТРОЙКИ

Файл `.env.example` показывает заглушки:
```env
BETTER_AUTH_SECRET="your-secret-key-here"
```

**Нужно:**
- Сгенерировать реальный секрет: `openssl rand -base64 32`
- Настроить OAuth провайдеры (GitHub, Google) — сейчас их нет
- Без HTTPS Better-auth cookie не будут работать (но Vercel даёт HTTPS автоматически)

---

### 4. **Нет реального пользователя** 🟡 ЗАМЕТНО

В `src/app/page.tsx`:
```typescript
const [currentUserId] = useState(() => `user_${Date.now()}`); // TODO: заменить на реальную авторизацию
```

Клиент генерирует случайный ID вместо реальной авторизации. Таблица лидеров будет показывать ботов и фейковых юзеров.

---

### 5. **Нет WebSocket/SSE для синхронизации** 🟡 СРЕДНЕЕ

Сейчас сигнал отправляется через tRPC (HTTP). Для реального времени между 2-5 игроками это работает, но задержка будет выше, чем с WebSocket/SSE.

---

### 6. **Нет мониторинга** 🟢 НИЗКИЙ ПРИОРИТЕТ

Нет Sentry, UptimeRobot, или любой системы отслеживания ошибок и аптайма.

## 📋 Ответ на вопрос

> **Есть ли в проекте всё необходимое для выполнения финального деплоя?**

**Нет, не всё.** Проект находится на уровне **~75% готовности к деплою**.

### Что нужно сделать перед деплоем:

| # | Задача | Приоритет | Сложность |
|---|--------|-----------|-----------|
| 1 | Оптимизировать БД-пул для serverless | 🔴 Критично | Низкая |
| 2 | Создать CI/CD (GitHub Actions) | 🔴 Критично | Низкая |
| 3 | Сгенерировать BETTER_AUTH_SECRET | 🟡 Средний | Низкая |
| 4 | Настроить OAuth провайдеры (GitHub/Google) | 🟡 Средний | Средняя |
| 5 | Подключить реальную авторизацию вместо `user_${Date.now()}` | 🟡 Средний | Средняя |
| 6 | Настроить Sentry для мониторинга | 🟢 Низкий | Низкая |
| 7 | (Опционально) WebSocket/SSE для синхронизации | 🟢 Низкий | Высокая |

### Что нужно для внешнего сервера:

**Да, нужен внешний хостинг.** Проект не может быть деплоинут "в пустоту" — нужны:

| Ресурс | Решение | Стоимость |
|--------|---------|-----------|
| **Хостинг** | Vercel (бесплатный план) или Railway | $0-25/мес |
| **PostgreSQL** | Neon (бесплатный план, serverless) | $0-20/мес |
| **Redis** | Upstash (бесплатный план) | $0-10/мес |
| **Домен + HTTPS** | Автоматически через Vercel | Бесплатно |
| **Мониторинг** | Sentry (бесплатный план) | $0-25/мес |

Всё это можно настроить за 1-2 дня без написания дополнительного кода (кроме оптимизации БД-пула).

---

**Итог:** Кодовая база проекта в целом готова к деплою. Основные препятствия — инфраструктурные (БД-пул, CI/CD), а не архитектурные. Внешний сервер **необходим** — проект не self-contained, он зависит от PostgreSQL и Redis.
